7. ARCHITECTURE FOR NEW MODULES
================================================================================

Module: beads_bridge.py
  Purpose: Bridge between Know graph and Beads project system
  Dependencies: GraphManager, EntityManager, DependencyManager
  Responsibilities:
    - Authenticate with Beads API
    - Transform data formats (graph ↔ Beads)
    - Handle network errors
    - Track sync history
  NOT Responsible For:
    - Deciding what to sync (task_sync does this)
    - Conflict resolution (task_sync does this)
    - Individual entity operations (task_manager does this)

Module: task_manager.py
  Purpose: Manage task entity lifecycle and state
  Dependencies: GraphManager, EntityManager, DependencyManager, GraphValidator
  Responsibilities:
    - Create/update/delete task entities
    - Manage task status and state transitions
    - Link tasks to features and other tasks
    - Define task types and state machines
  NOT Responsible For:
    - Syncing with external systems
    - Conflict resolution

Module: task_sync.py
  Purpose: Orchestrate synchronization between systems
  Dependencies: GraphManager, TaskManager, BeadsBridge, GraphValidator
  Responsibilities:
    - Two-way synchronization (graph ↔ Beads)
    - Detect changes and conflicts
    - Apply conflict resolution strategies
    - Track sync history
  NOT Responsible For:
    - Individual entity operations (task_manager)
    - API calls (beads_bridge)

Initialization Order:
  1. GraphManager
  2. EntityManager, DependencyManager, GraphValidator (depend on GraphManager)
  3. TaskManager (depends on #1, #2)
  4. BeadsBridge (depends on #1, #2, #3)
  5. TaskSync (depends on #1, #3, #4, #2)

9. KEY DESIGN PATTERNS
================================================================================

1. Manager Pattern
   Each subsystem has a Manager class:
   - GraphManager, EntityManager, DependencyManager
   - AsyncGraphManager, SpecGenerator
   - LLMManager, TaskManager, BeadsBridge, TaskSync

2. Validator Pattern
   Validation separated from operations:
   - GraphValidator (comprehensive)
   - DependencyManager.validate_graph() (dependency-specific)
   - EntityManager.validate_entity() (entity-specific)

3. Provider Pattern (LLM)
   Abstract provider with multiple implementations:
   - LLMProvider (base)
   - Implementations for different backends
   - MockProvider for testing

4. Dependency Injection
   Managers injected via CLI context:
   ctx.obj['manager'] = ManagerClass(dependencies)

5. Caching Strategy
   - Thread-safe in-memory cache (GraphCache)
   - Automatic invalidation on writes
   - Modification time tracking for reload

6. Rules-Based Behavior
   - Types and dependencies in JSON rules
   - Rules loaded at startup
   - Enables multi-graph support (spec vs code)
