CORE DESIGN PRINCIPLES
================================================================================

1. THIN WRAPPER
   BeadsBridge is a subprocess wrapper, nothing more
   → Direct passthrough means users get full Beads immediately
   → No command validation (subprocess list mode prevents injection)
   → Simple, maintainable, replaceable

2. REFERENCE-BASED STORAGE
   Store task IDs in references.beads (not new entity type)
   → Reuses existing schema (zero graph changes)
   → Flat structure (no nested relationships per CLAUDE.md)
   → Future-proof (can add native tasks in Phase 2)

3. BEADS-FIRST CONFLICT RESOLUTION
   Beads status always overwrites graph status on sync
   → Beads is execution layer, graph is planning layer
   → Makes sync predictable and consistent
   → Users understand the hierarchy

4. EXPLICIT SYNC (NOT AUTO)
   Manual `know bd sync` command, not automatic hooks
   → MVP is simpler, users stay in control
   → Prevents surprise updates
   → Auto-sync can be added in Phase 2 without changing design

5. NO EXTERNAL DEPENDENCIES
   Use only Python stdlib (subprocess, json, pathlib)
   → Lighter installation, no version conflicts
   → Simpler testing (all mocks in test code)
   → More portable (works anywhere Python works)

6. FAIL-FAST ERROR HANDLING
   Never silently fail - show clear error messages
   → bd not found? Show install instructions
   → Sync fails? Report what went wrong
   → Config invalid? Validate and error out
   → Users understand state of system

================================================================================
DATA FLOW EXAMPLE
================================================================================

Scenario: User adds feature, syncs with Beads

  $ know add feature auth "User authentication system"
    └─ EntityManager.add_entity('feature', 'auth', {...})
       └─ (optional) TaskSync.sync_feature_to_beads('feature:auth')
          └─ BeadsBridge.create_task('Auth', 'User authentication...')
             └─ subprocess.run(['bd', 'task-add', ...])
                └─ Beads creates task bd-a1b2
          └─ Store in references.beads['auth-a1b2']
             └─ GraphManager.save_graph()

  $ know bd sync
    └─ TaskSync.sync_all(trigger='explicit')
       ├─ sync_feature_to_beads() - Ensure all features have tasks
       │  └─ BeadsBridge.create_task() for missing ones
       │
       └─ sync_beads_to_graph() - Update status from Beads
          └─ BeadsBridge.list_tasks()
             └─ For each task linked to feature:
                ├─ Read status from Beads
                ├─ Compare with references.beads
                └─ Update if different, save graph

Result:
  ✓ Feature linked to Beads task (reference stored)
  ✓ Status synced from Beads (Beads is source of truth)
  ✓ Can use both systems in parallel

RISK ANALYSIS
================================================================================

LOW RISK (95% confidence):
  • Subprocess pattern - well-understood, no shell injection
  • Reference schema - uses existing section, backward compatible
  • GraphManager - read-only except save (already used)
  • CLI pattern - uses existing Click decorators
  → No breaking changes, safe to implement

MEDIUM RISK (manageable):
  • Task ID parsing - depends on Beads CLI format
    Mitigation: Extract via pattern matching, graceful fallback
  • Config loading - optional, uses safe defaults
    Mitigation: Non-blocking if config missing
  • Subprocess timeout - what if bd hangs?
    Mitigation: 30-second timeout kills process
  → All mitigated, acceptable for MVP

OVERALL RISK: LOW (95% confidence in success)
