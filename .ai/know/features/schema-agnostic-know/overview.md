# Feature: Schema-Agnostic Know

## User Request

Enable know CLI to work with custom schemas generated by LLMs via `/know:schema` command, allowing know to model any domain (conversations, research, habits, etc.) instead of being hardcoded for product development.

## Context

Currently, know uses hardcoded dependency rules from `config/dependency-rules.json` and `config/code-dependency-rules.json`. The `/know:schema` command generates custom schemas, but know can't validate against them yet.

## Requirements

### Core Functionality

1. **Schema Loader**
   - Detect schema format (new schema vs legacy dependency-rules)
   - Normalize both formats to common interface
   - Load schema from `meta.schema_path` in graph

2. **Field Validation**
   - Validate required fields when adding entities
   - Validate optional fields are allowed
   - Reject unknown fields

3. **Backward Compatibility**
   - Existing graphs with dependency-rules.json continue to work
   - No breaking changes to API
   - Support both schema formats simultaneously

4. **Auto-Detection**
   - Load schema from graph's `meta.schema_path`
   - Fall back to existing auto-detection (code-graph vs spec-graph)
   - Support explicit `-r` override

### Use Cases

1. **Conversation Memory**
   - Track conversations, people, concepts, decisions, preferences
   - Schema: conversation → person, concept, decision

2. **Research Notes**
   - Track papers, concepts, authors, citations
   - Schema: paper → concept, author, citation

3. **Habit Tracking**
   - Track habits, triggers, outcomes, streaks
   - Schema: habit → trigger, outcome, streak

4. **Project Management**
   - Track tasks, milestones, dependencies, people
   - Schema: task → milestone, person, dependency

## Success Criteria

- [ ] Know validates against custom schemas generated by `/know:schema`
- [ ] Field validation enforces required/optional fields
- [ ] Existing graphs with legacy dependency-rules.json work unchanged
- [ ] Graphs can reference schema via `meta.schema_path`
- [ ] All validation errors are clear and actionable
- [ ] Documentation includes examples of custom schemas

## Out of Scope

- Schema versioning and migration (future enhancement)
- Schema visualization tools
- Multi-schema graphs
- Schema inheritance

## Technical Constraints

- Must remain backward compatible (100%)
- No changes to graph file format (only additions)
- Existing know commands work unchanged
- Schema format should be extensible
